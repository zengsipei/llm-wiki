# Claude.md 内容筛选原则

## 核心结论

Claude.md（以及同类 AI agent 的持久上下文文件）应基本只保留**"环境信息"**——但这里的"环境信息"不是字面上的环境变量或系统配置，而是指**AI agent 无法从代码库或自身训练数据中直接推断出来的、项目特有的、稳定的规则**。

## 三条筛选标准

写入 CLAUDE.md 前必须同时满足以下三个条件：

### 1. 一眼看不出

Claude 通过阅读代码文件、目录结构、依赖列表、配置文件就能自然推断出的东西，不要写。例如：
- 项目用了 TypeScript + Next.js（`tsconfig.json`、`package.json` 已经说明）
- 目录结构（`ls` 一目了然）
- CLAUDE.md 本身的层级结构（`~/.claude/CLAUDE.md` → 根 → `.claude/rules/` → 子目录），这是 Claude Code 自身的机制，它的文档里写得很清楚

### 2. 不易变动

频繁变化的内容写在 CLAUDE.md 里会造成维护负担，且过时信息比没有信息更有害。例如：
- 当前正在开发的分支名、临时性的调试配置
- 依赖版本号（`package.json` 才是 source of truth）
- 暂时的架构决策（等稳定后再记）

### 3. 有实际使用价值（排除不常用的）

写了但不常用的规则会白白消耗每次会话的 token 注入成本。CLAUDE.md 的内容在每次会话启动时都会被注入到上下文中，每一行都有持续的成本。

## 删除测试法

判断一条规则是否值得保留的终极方法：**把它删掉，看看 Claude 会不会犯具体的错误**。如果删了也不会犯——说明 Claude 本身就能推断出来，不需要记。

## 应该保留的典型内容

- 项目特有的约定俗成（命名规范、提交格式偏好、分支策略）
- 非显而易见的架构决策及其原因（为什么选 A 不选 B）
- 跨文件的隐性依赖关系（代码里看不出来的耦合）
- 特定的测试运行方式或部署流程
- 团队独有的术语缩写或领域知识

## 不应该保留的典型内容

- 可以从代码/配置文件直接读出的信息
- 通用编程最佳实践（Claude 的训练数据已经覆盖）
- 临时性、频繁变动的信息
- Claude Code 自身机制的说明（它自己知道）
- 与当前项目无关的通用知识

## 本质

CLAUDE.md 的本质是**弥补 AI agent 的信息盲区**，而不是做项目的文档系统。它的受众是一个已经能读代码、能查文档的 AI，所以只需要提供那些"读了代码也看不出来的东西"。
