# Prompt Engineering 最佳实践

Prompt Engineering 是与大语言模型（LLM）高效交互的核心技能。本页面总结了经过验证的提示词工程技巧和模式。

## 基础原则

### 明确性

最好的 prompt 是明确的。模糊的 prompt 得到模糊的回答。具体来说：

- **做什么**：明确任务类型（总结、分析、翻译、编码）
- **怎么做**：指定输出格式、长度、语言、风格
- **不做什么**：设定边界（"不要编造信息"、"不要超过 500 字"）

### 角色设定

给 AI 一个明确的角色可以显著提升输出质量：

```
你是一个有 10 年经验的全栈工程师，擅长 TypeScript 和 React。
请帮我 review 这段代码，重点关注性能和类型安全。
```

角色设定通过以下机制生效：
- 激活模型中与该角色相关的知识领域
- 约束回答风格和专业深度
- 提供隐含的质量标准

## 高级技巧

### Chain-of-Thought（思维链）

让模型 "展示思考过程" 可以大幅提升推理准确度：

- 简单版：在 prompt 末尾加 "请一步步思考"
- 完整版：提供推理框架，让模型按步骤回答
- 自我验证：让模型先给出答案，再自我检查

### Few-Shot Learning（少样本学习）

提供 2-3 个示例比写长篇规则说明更有效：

```
任务：将以下产品描述改写为简洁的广告语

示例1：
输入："我们的产品是一个基于云端的团队协作工具，支持实时编辑..."
输出："云端协作，从此不再等"

示例2：
输入："这款手机搭载了最新的 A17 芯片，拥有 4800 万像素摄像头..."
输出："A17 芯片加持，每一拍都是大片"

现在请改写：
输入："我们的项目管理软件可以帮助团队更好地跟踪任务进度..."
```

### 分治策略

复杂任务拆解为子任务，每步只做一件事：

1. 第一步：分析需求，输出结构化大纲
2. 第二步：根据大纲逐段填充内容
3. 第三步：审阅和优化

### 迭代优化

不要期望一次得到完美结果。建立反馈循环：

1. 初始 prompt → 初版输出
2. 指出具体问题 + 要求修改 → 二版输出
3. 对比两版，提取最佳部分 → 最终版本

## 输出控制

### 格式约束

```
请按以下 JSON 格式输出：
{
  "title": "页面标题",
  "summary": "100字以内的摘要",
  "key_points": ["要点1", "要点2", "要点3"],
  "confidence": 0.85
}
```

### 长度控制

- "用不超过 200 字回答"
- "详细展开，至少 1000 字"
- "一句话总结"

### 语气控制

- 正式：用于文档、报告
- 口语：用于聊天、解释
- 学术：用于论文、研究
- 幽默：用于创意内容

## 常见反模式

| 反模式 | 问题 | 解决方案 |
|--------|------|----------|
| 过于简短 | "帮我写个登录页面" | 说明技术栈、风格、功能需求 |
| 过于冗长 | 大段背景描述 | 结构化：目标→约束→输出格式 |
| 自相矛盾 | "详细说明" + "简洁回答" | 统一要求，避免冲突指令 |
| 缺少上下文 | 直接给代码让改 | 说明项目背景、当前问题和期望行为 |
| 忽略输出格式 | 只说 "帮我分析" | 指定模板：先总结→再分析→最后给建议 |

## 工具辅助

- **Prompt 模板库**：将常用 prompt 存为模板，按场景调用
- **A/B 测试**：同一个任务用不同 prompt，对比结果
- **Token 计算**：长 prompt 注意控制长度，避免超出上下文窗口
- **版本管理**：将有效的 prompt 纳入版本控制，跟踪优化历程
