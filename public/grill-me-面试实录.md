## Q6：用户进度自提交设计

这个设计选择引起了我的注意。你提到 Go 的 WebSocket 只管终端交互，进度是用户自己提交给 PHP 的。我想追问两个层面：

1. 为什么不是自动检测？比如学员在终端里执行了正确命令、跑通了脚本，理论上 Go 服务可以检测到然后回调 PHP 标记完成。是故意设计成用户手动提交的吗？出于什么考虑？

**我：** 1. 设计如此，终端章节也是章节，所有章节，统一用户点击【提交、确定、下一个】，同时还避免了数据流转问题

2. 测试阶段就已经遇到这种问题了，最开始是想忽略掉不匹配进度，后来觉得都改版了，这种就是脏数据，直接清理掉

## Q7：支付系统

**面试官：** 很好。第一点说明你在设计上有统一的抽象思维——终端章节和普通章节对用户来说流程一致，不引入额外认知负担，同时跨服务不传数据，降低了耦合风险。第二点也说明你对数据质量有判断力，不是什么数据都迁就。

你提到负责了支付功能。电商和卖课场景下支付是核心链路，这个话题面试几乎必问。

1. 接了哪些支付渠道？支付宝、微信？有没有遇到"回调丢失"或"重复回调"的问题？你是怎么处理幂等的？
2. 课程购买涉及到一个有意思的点——用户付费后需要触发"创建快照队列"这个动作。如果支付回调了但创建快照失败了，或者反过来，你怎么保证这两步的原子性？用了事务、补偿机制、还是最终一致性？
3. 有没有遇到过"用户付了钱但课程没到账"的客诉？如果有，排查过程是怎样的？最终怎么修复的？

**我：** 1. 接入了支付宝、微信官方，同时还接入了一个第三方聚合支付的平台，中金支付。以前就遇到过回调丢失问题，因此这次是选择回调只是额外保障，主要还是主动查询。同时回调中除了写入事件日志，就是一个入异步队列，没有太多的业务逻辑，很快就回执了，这里你说的幂等是指哪方面？

2. 遇到了这个点，用的是补偿机制，最终一致。首先支付就是支付，不应该和其它的业务逻辑一起事务，其次，用户购买后到进入课程，还有一些链路，有时间补偿

3. 有一个点是，课其实和商品对应，几乎可以说是一条数据库记录，因此在回调+主动查询的模式下，没有漏过。课程内容才是创建快照队列，比较重的地方，确实有遗漏过，主要排查回调发送的异步队列消费情况，手动处理，后续也加入了类似主动查询，在用户进入课程页的时候，主动校验数据

**面试官（补充说明）：** 先回答你关于幂等的疑问——回调幂等是指：同一个支付通知被发送多次时，系统不能重复处理。比如支付宝可能会重试回调，如果你在回调里直接"创建订单+标记已付"，重试一次就创建了两条记录。你把回调只做"写入日志+入队列"，队列消费端通过订单状态判断是否已处理，这本身就是幂等做法。所以其实你已经做了，只是没有刻意叫这个名字。

## Q8：异步队列

**面试官：** 你在回答里多次提到了"异步队列"——回调入队列、支付后链路走队列。我想了解一下：

1. 用的什么队列？Redis 队列？还是消息中间件（RabbitMQ/Kafka）？选型理由是什么？
2. 队列消费失败怎么处理？有重试机制吗？重试几次？消费失败的消息是丢弃、死信队列、还是告警人工介入？
3. 有没有遇到过队列积压的情况？比如支付回调突然涌入、或者某个消费者变慢导致消息堆积？你是怎么发现的、怎么处理的？

**我：** 1. redis 队列。几乎没有学习成本，对团队友好，缺点是依赖的 swoole timer 来消费，只能秒级，但是对于公司够用了

2. 消费失败做了错误日志，邮件警告，有一个死信队列计数，在多次失败后，会从消费队列移除

3. 只遇到过日志队列积压的情况，是 redis 服务器内存告警，有写定时系统监控，超过阈值发邮件，云服务商那里也设置了短信告警之类，处理方式除了加大消费，就是丢弃部分日志。

## 面试官总评

先说结论：你对自己的评价，和你在这次面试里展示出来的水平，差距很大。

你在9个问题里的回答，我帮你梳理一下你"不经意"展示出来的东西：

**架构设计能力**
- 快照队列——这不是打螺丝，这是一个正经的领域建模决策。你识别到了"课程结构会变，但学员进度不能跟着变"这个核心矛盾，然后设计了一个购买时冻结快照的方案。面试官听到这个会眼前一亮。
- 无限级分类 + path 字段展平——说明你理解树形结构的多种实现方式，并且能根据业务场景选型。

**分布式系统思维**
- 支付回调只管接收入队列，主动查询兜底——这是标准的可靠消息模式
- 补偿机制 + 最终一致性，而不是大事务——说明你理解分布式事务的本质取舍
- 进度校验放在用户进入课程页时主动触发——这是典型的读时补偿

**工程成熟度**
- Redis 队列够用就不上 Kafka——知道选型要看场景
- 日志队列积压直接丢弃——能分清关键链路和非关键链路
- Composer 说服不了就不硬刚——知道什么时候该坚持、什么时候该务实
- 向量知识库做了成本评估后接受放弃——理解工程决策不只有技术维度

**你叫这些"打螺丝"，但这些是很多5年经验的人都说不清楚的。**

真正的问题不是你技术不行，是你讲故事的方式不行。

你说"没有太多作为工程师的感觉"——但你想想，如果面试的时候你把上面这些点用这种方式讲出来：

> "我主导设计了课程学习系统的核心模块。课程结构会动态调整，但学员进度不能变，所以我设计了一个快照队列机制——用户购买时把课程结构冻结一份，学员的学习进度基于这个快照推进。分类用的是无限级 path 展平设计，解锁条件通过快照内的数据判断。支付系统用了回调+主动查询双保障，业务逻辑走异步队列和补偿机制实现最终一致性。"

这段话，60秒说完，面试官对你的判断会完全不同。

你的问题是：你把这些当"日常工作"，没有意识到它们是卖点。
